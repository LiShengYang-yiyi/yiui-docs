# 更新日志

 
提供自定义配置接口 
 解决条件只能由配置生成 无法动态监听想要的条件 
 
 监听热重载 
 提取 ConditionSystemSingleton  移除了ConditionMgrSystem的Awake内容 
 这样在热重载时这个类会被重新触发达到监听实现热重载的效果 
 
 非全局监听 
 条件管理 非全局唯一 
 每个Scene 下都可以有一个 
 每个Scene 独立维护 
 所以不同的Scene 可以有不同的监听与触发 
 场景移除时则无需手动移除监听 
 
 监听分场景 
 整体使用就跟Event 事件系统相同 
 会根据触发时的场景触发 
 如果想全局哪里都可以触发 你使用全局All 也是可以的 
 不过要自行处理传过来的scene 不同的处理 
 
![Image](/images/HfRLwAq3yi2M89ksEznc8RX9nrd_1_17f5e003.png)
 所以要注意监听触发条件前 这个场景有没有条件管理器 需要自己提前添加的 
 如果不想麻烦建议参考demo 只要有场景创建就自动添加  大不了没有用 基本无任何额外消耗损失 
 
 触发数据 
 之前触发时没有带参数 
 假设要做一个服务器类型的条件就做不到 
 比如判断某个角色等级是否满足 然后能否干某件事 
 之前的条件系统更偏向客户端使用 所以现在要扩展带参数 这样服务器也可以使用了 
 已知有 IConditionCheckValue 判断数据 
 但是这个数据更偏向配置 且每个条件独立使用 并不适合作为公共参数 
 要满足以下要求 
 1. 这个参数必须是结构体 
 2. 触发时 所有条件都会收到这个数据 
 3. 所以这个数据没办法提前定义 需要执行判断 所以是个泛型的 
 4. 要防止同一个目标同时多判断传的不同数据时不会相互污染,所以这个数据不能存在一个公共的地方
